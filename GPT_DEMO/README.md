# GPT_DEMO：ChatGPT 快速可试玩草图阶段

## 定位

`GPT_DEMO/` 是正式 `WEB_DEMO/` 之前的快速验证区。

它用于支持制作人与 ChatGPT 在方案讨论阶段，快速生成可运行网页草图，让用户先直观看到方向是否一致。

核心目标不是工程质量，而是：

```text
快速看见；
快速试玩；
快速暴露偏差；
快速修正方向。
```

---

## 当前最新版本

```text
当前文件：GPT_DEMO/index.html
当前版本：gpt.8
当前状态：可试玩草图阶段
```

当前 Demo 用于验证：

```text
移动探索 → Space 近距离互动 → 收集辉石 → 派工人开路 → 点亮固定营地 → 固定黑雾口夜晚风险 → 营地矿山采矿 → 抵达阶段目标
```

---

## 如何运行

拉取仓库后，直接双击打开：

```text
GPT_DEMO/index.html
```

PC 操作方式：

```text
WASD / 方向键：移动主角
Space：与附近目标互动
R：重新生成 / 重新开始
```

移动端方向，暂未实现：

```text
横屏
左侧虚拟摇杆：上下左右移动
右侧点击 / 按钮：互动
```

---

## 与 WEB_DEMO 的区别

| 目录 | 作用 | 质量要求 |
|---|---|---|
| `GPT_DEMO/` | 方案讨论阶段的快速可试玩草图 | 可以粗糙，可以硬编码，重点是直观 |
| `WEB_DEMO/` | 正式 Web Demo 原型工作区 | 需要任务卡、结构清晰、可维护、可配置 |

## gpt.8：固定黑雾口与营地矿山经营

### 修改内容

- 地图生成固定“黑雾口”。
- 夜晚怪物只从黑雾口出现，不再从目标附近随机生成。
- 白天黑雾口作为危险提示存在。
- 怪物继续遵循抢夺辉石逻辑：优先抢散落辉石，其次威胁安全区外的玩家和工人。
- 点亮旧火塘后，会在附近生成一个矿山。
- 玩家靠近矿山按 Space，可派最近可调度工人去采矿。
- 工人进入矿山后变为 `mining` 状态。
- `mining` 工人不参与砍树、修桥、点亮旧火塘等普通任务调度。
- 矿山每 10 秒产出 1 个单颗辉石。
- 玩家靠近产出的辉石后按 Space 拾取。
- HUD 增加黑雾口数量、当前怪物数量和采矿工人数显示。

### 修改原因

让夜晚风险从“随机出现”变为“可观察、可预判、可规划”的固定危险源；同时让营地从单纯安全点扩展为资源恢复点，验证“白天远征、夜晚经营”的双循环雏形。

### 遗留问题

- 矿山工人暂不支持召回。
- 矿山没有产出上限。
- 黑雾口位置和数量仍需试玩调优。
- 怪物返回黑雾口的动画暂未实现，当前捡到辉石后直接消失。
- 移动端虚拟摇杆和右侧互动按钮尚未实现。

# GitHub 写入异常处理经验

## 现象

后续测试发现：

1. `.md` 文件可以正常写入 `main`。
2. `GPT_DEMO/index.html` 可以在开发分支写入。
3. `GPT_DEMO/index.html` 直接写入 `main` 时，可能被 OpenAI 安全检查拦截。
4. 频繁写入、反复更新同一个 HTML / JS 文件、大段代码写入、直接修改 `main` 上的可执行网页文件，可能提高触发概率。

因此，该问题更像是 ChatGPT / OpenAI 工具层的安全检查，而不一定是 GitHub 仓库权限问题。

## 判断方法

当后续再次遇到 GitHub 写入异常时，按以下顺序排查：

1. 先测试是否能读取仓库。
2. 再测试是否能写入一个小 `.md` 文件。
3. 如果 `.md` 也不能写，优先提示用户断开 GitHub 授权，然后重新授权 GitHub 连接器。
4. 如果 `.md` 能写，但 `GPT_DEMO/index.html` 写入 `main` 被拒绝，说明 GitHub 授权大概率正常，问题集中在主干 HTML / JS 写入安全检查。
5. 如果开发分支可以写入 `index.html`，则说明可以通过分支 + PR 方式绕过直接写入主干的问题。

## 后续固定写入流程

为了减少风险，后续采用以下流程：

1. 先在对话中集中讨论 2–4 个功能，形成一个“小版本包”。
2. 先整理版本目标、修改点和验收标准。
3. 用户确认后，再尝试写入 GitHub。
4. 默认先尝试直接写入 `main`。
5. 如果 `main` 写入成功，则完成提交。
6. 如果 `main` 写入失败或被安全检查拦截，则改用：
   - 从 `main` 创建开发分支；
   - 在开发分支写入 `GPT_DEMO/index.html`；
   - 创建 PR；
   - 用户确认后再合并到 `main`。
7. 如果连 `.md` 文件也不能写入，则提示用户断开并重新授权 GitHub。

## 分支策略

- `main`：稳定试玩版本。
- `dev/gpt-x` 或类似开发分支：GPT_DEMO 新版本验证。
- 版本说明优先写入已有文档，例如 `GPT_DEMO/README.md`；除非用户明确同意，不新增零散记录文件。

## gpt.11.4：轻量自动战斗与围墙系统（待开发提示词）

请修改 `GPT_DEMO/index.html`，并同步更新 `GPT_DEMO/README.md` 的版本记录。不要新增文件，不要自动提交 GitHub。

### 版本目标

在当前 gpt.11.3 的基础上，加入一个类似《Kingdom New Lands》的轻量自动战斗验证系统。玩家不直接攻击，玩家通过辉石、工人、弓箭手、围墙和营地布置来影响夜晚防守结果。

### 核心设计原则

1. 不做玩家手动攻击。
2. 不做复杂动作战斗。
3. 玩家负责资源分配、建造、转职、放置辉石、营地防守准备。
4. 弓箭手负责自动攻击黑影。
5. 黑影负责夜晚推进、拾取辉石、攻击围墙和角色。

### 规则一：黑影不再因为主角进入安全光圈而直接消失

当前问题：部分逻辑可能在黑影失去可攻击角色、目标进入安全区或夜晚结束时直接让黑影消失。

新规则：

1. 黑影不应因为主角进入部落/营地光圈就直接消失。
2. 黑影应持续存在并重新寻找目标。
3. 黑影每帧按 gpt.11.3 的优先级寻找目标：
   - 第一优先：附近 8×8 范围内的辉石。
   - 第二优先：围墙。
   - 第三优先：角色，顺序为 工人 > 弓箭手 > 返程流民 > 主角。
4. 如果没有任何目标，黑影返回最近黑雾口。
5. 黑影抵达黑雾口后消失。
6. 夜晚结束后，黑影也应尝试返回黑雾口，而不是原地直接消失。

### 规则二：黑影 HP

1. 所有新生成黑影增加 `hp` 字段。
2. 普通黑影 `hp = 1`。
3. 弓箭手射中黑影时，黑影 `hp -= 1`。
4. `hp <= 0` 时黑影消失。
5. 当前版本暂不做不同黑影类型，但字段要为后续扩展保留。

### 规则三：弓箭手自动战斗

将当前弓箭手“直接驱散黑影”的临时逻辑升级为正式自动战斗：

1. 夜晚时，弓箭手自动搜索射程内黑影。
2. 射程建议为 `5.5` 格。
3. 弓箭手进入瞄准状态，瞄准时间建议为 `0.6` 秒。
4. 瞄准完成后射击。
5. 一箭伤害为 `1`。
6. 射击后进入冷却，冷却建议为 `2.2` 秒。
7. 当前版本命中率为 `100%`，不要做随机命中率。
8. 弓箭手显示状态：`瞄准`、`射击`、`冷却`、`弓箭手`。
9. 只要黑影还活着，弓箭手可继续射击。

### 规则四：围墙建造

新增最小围墙系统，用于验证营地防守策略。

1. 玩家可以在营地附近建造围墙。
2. 建造方式建议：玩家靠近营地或旧火塘附近的可建造点，按 Space 派工人建造。
3. 为降低开发量，可以先在每个部落/营地左右两侧自动生成两个“围墙地基”。
4. 围墙地基显示为 `墙基`。
5. 玩家对墙基按 Space，消耗 `2` 个辉石，派最近可用工人建造围墙。
6. 工人完成建造后，墙基变为 `围墙`。
7. 围墙拥有 `hp = 3`。
8. 围墙可以被黑影攻击。
9. 黑影攻击围墙一次，围墙 `hp -= 1`。
10. 围墙被攻击三次后消失，地块恢复为普通地面或破墙状态。
11. 围墙显示剩余耐久，例如 `围墙 3/3`、`围墙 2/3`、`围墙 1/3`。

### 规则五：黑影与围墙交互

1. 黑影目标优先级变为：
   - 附近辉石。
   - 阻挡其前进或附近的围墙。
   - 角色目标：工人 > 弓箭手 > 返程流民 > 主角。
   - 无目标时返回黑雾口。
2. 如果黑影目标是围墙，黑影靠近后进入攻击状态。
3. 黑影每隔 `1.0` 秒攻击一次围墙。
4. 围墙 `hp <= 0` 后消失。
5. 围墙消失后，黑影重新选择目标。

### 建议地块类型

如当前 `T` 枚举还有空间，可以新增：

```js
WALL_BASE: 17,
WALL: 18,
```

如果不想改枚举名字，也可以使用当前风格：

```js
WB: 17,
WL: 18,
```

### 建议实现细节

1. 新增 `spawnWallBases(cx, cy)`：在部落/营地附近生成两个墙基。
2. 初始部落生成时调用一次。
3. 新营地建成时也调用一次。
4. `infoAt()` 增加墙基互动：`建造围墙 2`。
5. `finish(w)` 增加 `wall` 任务完成逻辑。
6. `drawTile()` 增加墙基和围墙显示。
7. `findNearestTarget(m)` 增加围墙目标分支。
8. `updateMonsters(dt)` 增加围墙攻击逻辑。
9. `spawnMonsters()` 新生成黑影时设置 `{ x, y, hp: 1, atk: 0, returning: false }`。

### 验收标准

1. 页面能直接打开，浏览器控制台没有 JavaScript 报错。
2. 夜晚黑影不会因为主角进入安全光圈就直接消失。
3. 黑影没有目标时会返回黑雾口并消失。
4. 弓箭手会自动瞄准并射击射程内黑影。
5. 黑影拥有 hp，弓箭手射中后 hp 减少，hp 为 0 时黑影消失。
6. 玩家可以建造围墙。
7. 围墙拥有 3 点耐久。
8. 黑影会攻击围墙。
9. 围墙被攻击 3 次后消失。
10. 围墙能让玩家感受到营地防守价值。
11. 黑影仍然优先拾取附近辉石。
12. 没有辉石和围墙时，黑影按 工人 > 弓箭手 > 返程流民 > 主角 攻击角色。
13. README 新增 `gpt.11.4：轻量自动战斗与围墙系统` 版本记录，写清修改内容、修改原因、验收标准和遗留问题。

1. 不做玩家手动攻击。
2. 不做多种敌人。
3. 不做城墙升级。
4. 不做弓箭手命中率。
5. 不做复杂弹道。
6. 不做建筑 UI 面板。
