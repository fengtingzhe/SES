# GPT_DEMO：ChatGPT 快速可试玩草图阶段

## 定位

`GPT_DEMO/` 是正式 `WEB_DEMO/` 之前的快速验证区，用于在方案讨论阶段快速生成可运行网页草图，让制作人先直观看到方向是否一致。

核心目标：

```text
快速看见；
快速试玩；
快速暴露偏差；
快速修正方向。
```

---

## 当前最新版本

```text
当前文件：GPT_DEMO/index.html
当前版本：gpt.11.7
当前状态：可试玩草图阶段
```

当前 Demo 用于验证：

```text
探索 → 辉石收集/放置 → 工人派遣 → 砍树/修桥/建营地 → 流民转职 → 弓箭手防守 → 黑影夜袭/攻城 → 围墙防御 → 流民火堆刷新 → 奇遇/特殊地形 → 抵达阶段目标
```

---

## 如何运行

拉取仓库后，直接双击打开：

```text
GPT_DEMO/index.html
```

PC 操作方式：

```text
WASD / 方向键：移动主角
Space：与面前或附近目标互动；无互动目标时放置辉石
R：重新生成 / 重新开始
F2：开关文字标签
F3：开关小地图
```

---

## 与 WEB_DEMO 的区别

| 目录 | 作用 | 质量要求 |
|---|---|---|
| `GPT_DEMO/` | 方案讨论阶段的快速可试玩草图 | 可以粗糙，可以硬编码，重点是直观 |
| `WEB_DEMO/` | 正式 Web Demo 原型工作区 | 需要任务卡、结构清晰、可维护、可配置 |

## gpt.11.4：轻量自动战斗与围墙系统

### 修改内容

- 黑影不再因为主角进入安全光圈而直接消失。
- 黑影无目标或夜晚结束时，返回最近黑雾口，抵达后消失。
- 黑影新增 `hp` 字段，普通黑影 `hp=1`。
- 弓箭手升级为自动战斗单位：瞄准 0.6 秒，射击造成 1 点伤害，冷却 2.2 秒。
- 新增墙基 / 围墙地块。
- 部落和新营地附近生成墙基。
- 玩家对墙基按 Space，消耗 2 个辉石，派工人建造围墙。
- 围墙 `hp=3`，黑影每 1 秒攻击一次围墙，围墙被攻击 3 次后消失。
- 黑影目标优先级为：辉石 > 围墙 > 工人 > 弓箭手 > 返程流民 > 主角。

### 修改原因

让夜晚不只是逃跑和丢辉石，而是形成“防守准备 → 黑影推进 → 弓箭手射击 → 围墙吸收伤害”的轻量自动战斗闭环，接近《Kingdom New Lands》的间接战斗体验。

### 已知问题（已在 gpt.11.5 中修正）

- ~~围墙搜索范围偏小~~ → gpt.11.5 改为全地图搜索。
- ~~spawnMonsters 未纳入围墙~~ → gpt.11.5 新增 `hasWall()` 判断。
- 围墙仍在 `pass()` 可通行列表中，玩家和工人都能穿过围墙（暂不处理）。
- ~~流民火堆一次性失效~~ → gpt.11.5 新增 10 秒冷却刷新。

## gpt.11.5：黑影攻城与平衡修正

### 修改内容

- 黑影围墙搜索从 8×8 范围改为全地图搜索，确保黑影稳定攻城。
- 新增 `hasWall()` 辅助函数，`spawnMonsters()` 将围墙纳入可攻击目标判断。
- 黑影生成从随机概率改为每夜固定数量，默认 `MONSTERS_PER_NIGHT = 2`，加 3 秒生成间隔避免同帧刷怪。
- 新增 `nightSpawned` / `lastSpawnNightDay` / `updateNightSpawnCounter()` 每夜重置计数。
- 流民火堆招募后进入 10 秒冷却（`refCd`），冷却结束后自动恢复可招募。
- 新增 `updateRefugeeFires(dt)` 每帧更新冷却计时。
- 流民火堆标签显示冷却倒计时（`冷却 Xs`）。
- HUD 增加本夜黑影已生成数量显示（`本夜 N/2`）。

### 修改原因

gpt.11.4 中围墙搜索范围太小（8×8），导致黑影常常找不到围墙直接返回黑雾口，围墙形同虚设。随机刷怪概率难以控制平衡。流民火堆一次性招募后永久失效导致人口增长断掉。本次修正让攻城逻辑可预期、可规划、可防守。

### 验收标准

1. 页面能直接打开，浏览器控制台没有 JavaScript 报错。
2. 地图上存在围墙时，即使营地外没有角色和辉石，黑影也会向最近围墙前进。
3. 黑影不会因为围墙超出 8×8 范围就立刻返回黑雾口。
4. 黑影会攻击围墙，每 1 秒造成 1 点伤害，被攻击 3 次后消失。
5. 黑影每晚最多生成固定数量，默认 2 只。
6. HUD 能看出本夜已生成黑影数量（`本夜 N/2`）。
7. 流民火堆招募后进入 10 秒冷却，标签显示冷却倒计时。
8. 冷却结束后，流民火堆重新变为可招募（`流民火堆`）。
9. 冷却中的流民火堆不能被重复招募。
10. 原有功能不被破坏：Space 互动、Space 放置辉石、工人派遣、弓箭手射击、围墙建造、黑影拾辉石、黑影返回黑雾口、狐狸成亲、颠倒森林、F2/F3 性能诊断。

### 遗留问题

- 围墙仍在 `pass()` 可通行列表中，玩家和工人都能穿过围墙。
- 黑影返回黑雾口的动画较简单。
- 流民火堆冷却没有声音或其他明显提示。
- `MONSTERS_PER_NIGHT` 值需要继续试玩调优。

## gpt.11.6：黑影局部进攻逻辑

### 修改内容

- 黑影生成时，选择距离黑雾口最近的部落 / 营地作为 `raidCamp`。
- 黑影默认向 `raidCamp` 行军，不再全地图寻找最近围墙。
- 新增黑影局部感知范围 `MONSTER_TACTICAL_RANGE = 4`。
- 黑影只会被自身 4 格范围内的辉石、围墙、工人、弓箭手、返程流民和主角吸引。
- 新增目标锁定时间 `MONSTER_TARGET_LOCK = 1.2`，减少辉石反复丢弃 / 捡起造成的目标拉扯。
- `safe()` 不再用于黑影目标过滤，营地光圈不再提供绝对免疫。
- 工人即使在营地光圈内，也会感知黑影并撤退。
- HUD / 帮助文本 / 右侧版本面板同步更新到 gpt.11.6。

### 修改原因

gpt.11.5 的“全地图最近围墙”逻辑会导致黑影跨地图攻击部落围墙，而不是攻击距离黑雾口最近的新营地。该行为不符合玩家对“黑雾口威胁附近营地”的直觉。本次改为“战略目标 + 局部目标”的两层逻辑：黑影先锁定最近营地，再只在行进中响应附近 4 格内的目标。

### 验收标准

1. 页面能直接打开，浏览器控制台没有 JavaScript 报错。
2. 黑影从黑雾口生成后，会向距离该黑雾口最近的部落 / 营地行军。
3. 新营地离黑雾口最近时，黑影优先攻击新营地方向，不再跨地图打部落墙。
4. 黑影不再全地图搜索围墙。
5. 玩家远处反复丢 / 捡辉石，不会让黑影跨地图改变目标。
6. 玩家靠近黑影并在 4 格内放置辉石时，黑影会被局部辉石吸引。
7. 黑影接近营地后，如果 4 格内有围墙，则优先攻击围墙。
8. 如果 4 格内没有围墙但有工人、弓箭手、返程流民或主角，则按局部目标优先级攻击。
9. 主角 `pl.inv > 0` 时仍不会被攻击。
10. gpt.11.5 的固定刷怪、流民火堆刷新、弓箭手自动射击、辉石放置、狐狸成亲、颠倒森林、F2/F3 仍保留。

### 遗留问题

- 工人完成砍树后是否总能返回最近营地仍需实测验证。
- 采矿工人是否应该主动撤退暂不确定。
- 围墙仍是可攻击建筑，不是严格阻挡物。
- `hasWall()` 已成为旧版遗留工具函数，后续可考虑删除或继续保留为调试辅助。

## gpt.11.7：状态清理与夜晚刷怪修正

### 修改内容

- 修复矿山占用残留：`release(u)` 遍历全图清理 `mine.worker === u.id` 的矿山占用。
- `sendMine()` 增加保险清理：矿山记录的工人不存在或 `lost=true` 时自动清空 `mine.worker`。
- 修复连续夜晚跨天时刷怪计数重置问题：新增 `wasNight` 状态，`nightSpawned` 只在”从非夜进入夜”瞬间重置，不再按 `day` 变化判断。
- 删除 `lastSpawnNightDay`，以 `wasNight` 替代。
- 修复撤退工人可被重新派遣：`bestWorker()` 过滤条件增加 `!w.flee`，排除撤退中的工人。
- 页面版本展示、HUD、帮助文本、右侧版本面板同步更新为 gpt.11.7。

### 修改原因

- 避免矿工被黑影抓走后矿山仍提示”矿山已有工人”，导致矿山永久不可互动。
- 保证”每晚固定 2 只黑影”的规则稳定，不因夜晚跨过 `day++` 重复刷新一轮。
- 避免玩家误操作把正在撤退的工人重新派出去，破坏撤退与风险反馈。

### 验收标准

1. 页面能直接打开，浏览器控制台没有 JavaScript 报错。
2. 页面所有版本展示均为 gpt.11.7。
3. 矿工被黑影抓走后，矿山不再永久显示”矿山已有工人”，可重新派工。
4. 矿山 `mine.worker` 指向的工人不存在或 `lost=true` 时，再次互动矿山会自动清理占用。
5. 一个连续夜晚中，黑影最多生成 `MONSTERS_PER_NIGHT` 只，不因跨天重新生成一轮。
6. 撤退中的工人 `flee=true` 不会被 `bestWorker()` 选中重新派遣。
7. gpt.11.6 功能仍保留：最近营地行军、4 格局部感知、目标锁定、营地无免疫。
8. 原有功能不被破坏：Space 互动、Space 放置辉石、工人派遣、弓箭手射击、围墙建造、黑影拾辉石、黑影返回黑雾口、狐狸成亲、颠倒森林、F2/F3 性能诊断。

### 遗留问题

- 工人完成砍树后是否总能返回最近营地，暂时保留观察，后续实测后再决定是否修改 `homePath()` / `nearestHome()` / `homeX` / `homeY`。
- 采矿工人是否应该主动撤退，本版本暂不做复杂的”矿工撤退后恢复采矿”系统。
- 围墙目前仍是可攻击建筑，不是严格阻挡物。
- 流民火堆在冷却结束但流民仍返程时，可能出现”显示已刷新但仍被 lock 阻止互动”的体验困惑，后续可优化 UI 文案。

# GitHub 写入异常处理经验

## 现象

后续测试发现：

1. `.md` 文件通常可以正常写入 `main`。
2. `GPT_DEMO/index.html` 可以在开发分支写入。
3. `GPT_DEMO/index.html` 直接写入 `main` 时，可能被 OpenAI 安全检查拦截。
4. 频繁写入、反复更新同一个 HTML / JS 文件、大段代码写入、直接修改 `main` 上的可执行网页文件，可能提高触发概率。

## 后续固定写入流程

1. 先在对话中集中讨论 2–4 个功能，形成一个“小版本包”。
2. 先整理版本目标、修改点和验收标准。
3. 用户确认后，再尝试写入 GitHub。
4. 默认先尝试直接写入 `main`。
5. 如果 `main` 写入失败或被安全检查拦截，则把提示词写入 `GPT_DEMO/README.md`。
6. 如果连 `.md` 文件也不能写入，则把提示词直接给用户，由本地 AI / Codex 执行。
