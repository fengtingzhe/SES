# GPT_DEMO：ChatGPT 快速可试玩草图阶段

## 定位

`GPT_DEMO/` 是正式 `WEB_DEMO/` 之前的快速验证区，用于在方案讨论阶段快速生成可运行网页草图，让制作人先直观看到方向是否一致。

核心目标：

```text
快速看见；
快速试玩；
快速暴露偏差；
快速修正方向。
```

---

## 当前最新版本

```text
当前文件：GPT_DEMO/index.html
当前版本：gpt.11.4
当前状态：可试玩草图阶段
```

当前 Demo 用于验证：

```text
探索 → 辉石收集/放置 → 工人派遣 → 砍树/修桥/建营地 → 流民转职 → 弓箭手防守 → 黑影夜袭 → 围墙防御 → 奇遇/特殊地形 → 抵达阶段目标
```

---

## 如何运行

拉取仓库后，直接双击打开：

```text
GPT_DEMO/index.html
```

PC 操作方式：

```text
WASD / 方向键：移动主角
Space：与面前或附近目标互动；无互动目标时放置辉石
R：重新生成 / 重新开始
F2：开关文字标签
F3：开关小地图
```

---

| 目录 | 作用 | 质量要求 |
|---|---|---|
| `GPT_DEMO/` | 方案讨论阶段的快速可试玩草图 | 可以粗糙，可以硬编码，重点是直观 |
| `WEB_DEMO/` | 正式 Web Demo 原型工作区 | 需要任务卡、结构清晰、可维护、可配置 |

## gpt.11.4：轻量自动战斗与围墙系统

### 修改内容

- 黑影不再因为主角进入安全光圈而直接消失。
- 黑影无目标或夜晚结束时，返回最近黑雾口，抵达后消失。
- 黑影新增 `hp` 字段，普通黑影 `hp=1`。
- 弓箭手升级为自动战斗单位：瞄准 0.6 秒，射击造成 1 点伤害，冷却 2.2 秒。
- 新增墙基 / 围墙地块。
- 部落和新营地附近生成墙基。
- 玩家对墙基按 Space，消耗 2 个辉石，派工人建造围墙。
- 围墙 `hp=3`，黑影每 1 秒攻击一次围墙，围墙被攻击 3 次后消失。
- 黑影目标优先级为：辉石 > 围墙 > 工人 > 弓箭手 > 返程流民 > 主角。

### 修改原因

让夜晚不只是逃跑和丢辉石，而是形成“防守准备 → 黑影推进 → 弓箭手射击 → 围墙吸收伤害”的轻量自动战斗闭环，接近《Kingdom New Lands》的间接战斗体验。

### 已知问题

- 当前围墙搜索范围偏小，黑影只会找附近 8×8 范围内的围墙，导致黑影离营地较远时可能直接返回黑雾口。
- `spawnMonsters()` 没有把围墙算作生成黑影的目标条件。
- 围墙当前仍在 `pass()` 可通行列表中，更像“可攻击建筑”，还不是严格阻挡物。
- README 曾长期停留在旧版本，需要后续开发时继续保持同步更新。

# GitHub 写入异常处理经验

## 现象

后续测试发现：

1. `.md` 文件通常可以正常写入 `main`。
2. `GPT_DEMO/index.html` 可以在开发分支写入。
3. `GPT_DEMO/index.html` 直接写入 `main` 时，可能被 OpenAI 安全检查拦截。
4. 频繁写入、反复更新同一个 HTML / JS 文件、大段代码写入、直接修改 `main` 上的可执行网页文件，可能提高触发概率。

## 后续固定写入流程

1. 先在对话中集中讨论 2–4 个功能，形成一个“小版本包”。
2. 先整理版本目标、修改点和验收标准。
3. 用户确认后，再尝试写入 GitHub。
4. 默认先尝试直接写入 `main`。
5. 如果 `main` 写入失败或被安全检查拦截，则把提示词写入 `GPT_DEMO/README.md`。
6. 如果连 `.md` 文件也不能写入，则把提示词直接给用户，由本地 AI / Codex 执行。

### 一、修正黑影攻击围墙逻辑

当前问题：

```text
黑影寻找围墙时只搜索自己周围 8×8 范围。
如果营地外没有角色或辉石，而围墙距离黑雾口较远，黑影找不到围墙，就会立刻返回黑雾口。
```

正确规则：

```text
辉石仍然只作为附近诱饵，保持 8×8 感知范围。
围墙是营地防御目标，黑影应该全地图感知最近围墙。
只要地图上存在围墙，黑影就应优先攻击最近围墙，而不是返回黑雾口。
```

请修改 `findNearestTarget(m)`：

1. 第一优先级：附近 8×8 范围内的辉石。
2. 第二优先级：全地图最近围墙 `T.WL`，不再限制 8×8。
3. 第三优先级：工人。
4. 第四优先级：弓箭手。
5. 第五优先级：返程流民。
6. 最后：主角。
7. 如果以上目标都不存在，才返回 `null`。

围墙搜索参考逻辑：

```js
let walls = [];
for (let y = 0; y < H; y++) {
  for (let x = 0; x < W; x++) {
    let c = cell(x, y);
    if (c && c.t === T.WL && (c.hp || 0) > 0) {
      walls.push({ x, y, hp: c.hp });
    }
  }
}
if (walls.length) {
  walls.sort((a, b) => dist(m, a) - dist(m, b));
  return { kind: 'wall', obj: walls[0] };
}
```

### 二、spawnMonsters 增加围墙目标判断

当前问题：

```text
spawnMonsters() 只把安全区外角色和地图上的辉石当作可生成黑影的理由。
如果没有角色、没有辉石，但地图上有围墙，黑影可能不会生成或生成后立刻返回。
```

新增辅助函数：

```js
function hasWall() {
  for (let y = 0; y < H; y++) {
    for (let x = 0; x < W; x++) {
      let c = cell(x, y);
      if (c && c.t === T.WL && (c.hp || 0) > 0) return true;
    }
  }
  return false;
}
```

把原来的判断：

```js
if (!hasTarget && !hasStone) return;
```

改为：

```js
if (!hasTarget && !hasStone && !hasWall()) return;
```

### 三、黑影生成数量改为固定数量

当前问题：

```text
黑影使用随机概率生成，可能在某些夜晚刷得太多，导致平衡异常和 FPS 压力。
```

目标规则：

```text
每个夜晚只生成固定数量黑影。
建议初始值：每晚 2 只。
```

实现要求：

1. 新增全局变量，例如：

```js
let nightSpawned = 0;
let lastNightDay = 0;
const MONSTERS_PER_NIGHT = 2;
```

2. 每进入一个新的夜晚，重置 `nightSpawned = 0`。
3. `spawnMonsters()` 不再使用纯概率无限尝试刷怪。
4. 夜晚期间，如果当前夜晚已生成数量 `< MONSTERS_PER_NIGHT`，且存在可攻击目标，则生成黑影。
5. 每生成一只黑影，`nightSpawned++`。
6. 可以保留 `monsters.length` 上限，例如 `monsters.length > 5` 时不生成。
7. HUD 或 Console 中可以显示：`本夜黑影：nightSpawned / MONSTERS_PER_NIGHT`，便于验证。

伪代码：

```js
const MONSTERS_PER_NIGHT = 2;
let nightSpawned = 0;
let lastSpawnNightDay = 0;

function updateNightSpawnCounter() {
  if (night()) {
    if (lastSpawnNightDay !== day) {
      nightSpawned = 0;
      lastSpawnNightDay = day;
    }
  }
}
```

在 `update(dt)` 中调用：

```js
updateNightSpawnCounter();
```

在 `spawnMonsters()` 中：

```js
if (!night()) return;
if (nightSpawned >= MONSTERS_PER_NIGHT) return;
if (monsters.length > 5) return;
// 存在目标后生成
nightSpawned++;
```

如果想避免 2 只黑影同一帧同时生成，可以加一个 `spawnCooldown`：

```js
let spawnCooldown = 0;
```

每次生成后设为：

```js
spawnCooldown = 3;
```

每帧递减，只有 `spawnCooldown <= 0` 才能继续生成。

### 四、流民火堆每 10 秒可重新招募

当前问题：

```text
流民火堆招募一次后 ref=false，基本永久失效。
这会导致人口增长断掉，后续工人/弓箭手/围墙系统平衡异常。
```

目标规则：

```text
流民火堆被招募后进入冷却。
冷却时间 10 秒。
冷却结束后，该火堆重新变为可招募状态。
```

实现要求：

1. `T.RF` 火堆增加字段：

```js
ref: true / false
refCd: 0
```

2. 招募流民时：

```js
c.ref = false;
c.refCd = 10;
```

3. 新增 `updateRefugeeFires(dt)`：

```js
function updateRefugeeFires(dt) {
  for (let y = 0; y < H; y++) {
    for (let x = 0; x < W; x++) {
      let c = cell(x, y);
      if (c && c.t === T.RF && !c.ref && c.refCd != null) {
        c.refCd -= dt;
        if (c.refCd <= 0) {
          c.ref = true;
          c.refCd = 0;
        }
      }
    }
  }
}
```

4. 在 `update(dt)` 中调用：

```js
updateRefugeeFires(dt);
```

5. `drawTile()` 对流民火堆显示：

```text
可招募：流民火堆
冷却中：流民火堆 8s
```

6. `infoAt()` 中，只有 `c.t === T.RF && c.ref === true` 时才允许互动。
7. 玩家靠近冷却中的流民火堆按 Space，不应招募成功；可选提示：`流民火堆还在冷却。`

### 七、验收标准

1. 页面能直接打开，浏览器控制台没有 JavaScript 报错。
2. 地图上存在围墙时，即使营地外没有角色和辉石，黑影也会向最近围墙前进。

5. 围墙被攻击 3 次后消失。
6. 黑影每晚最多生成固定数量，默认 2 只。
7. HUD 或 Console 能看出本夜已生成黑影数量。
8. 流民火堆招募后进入 10 秒冷却。
9. 冷却结束后，流民火堆重新可招募。
10. 冷却中的流民火堆不能被重复招募。
11. 原有功能不被破坏：Space 互动、Space 放置辉石、工人派遣、弓箭手射击、围墙建造、黑影拾辉石、黑影返回黑雾口、狐狸成亲、颠倒森林、F2/F3 性能诊断。
